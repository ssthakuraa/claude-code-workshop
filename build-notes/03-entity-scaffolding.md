# Build Note: Entity Scaffolding Pattern

**Date:** 2026-03-30
**Module:** Backend — Entity scaffold (Region as the template)
**Maps to Lab:** Lab 2: Skills

---

## What We Built

A repeatable scaffolding pattern for every HR entity, using `Region` as the first example:
- JPA Entity (`HrRegion`) with `@SQLRestriction` soft-delete
- Repository (`HrRegionRepository`)
- DTOs: `HrRegionDTO`, `HrRegionRequest`
- Service (`HrRegionService`) with full `HrLogHelper` entry/exit logging
- Controller (`HrRegionController`) returning `HrApiResponse<T>`

Once the pattern was stable, all other simple entities (Country, Location, Job, Department) were generated by repeating it.

## Technique Used

**Skills** — created a `/scaffold-entity` skill after confirming the Region pattern worked. All subsequent entities used the skill rather than re-prompting from scratch.

## The Prompt That Worked

```
Scaffold the Region entity following CLAUDE.md conventions:
1. HrRegion entity → regions table (regionId, regionName, deletedAt)
   @SQLRestriction("deleted_at IS NULL"), @SoftDelete pattern
2. HrRegionRepository extends JpaRepository
3. HrRegionDTO (output), HrRegionRequest (input, @NotBlank on name)
4. HrRegionService: findAll, findById, create, update, delete (soft)
   Full HrLogHelper entry/exit on every method. Never log PII.
5. HrRegionController: GET /, GET /{id}, POST, PUT /{id}, DELETE /{id}
   All return HrApiResponse<T>. Base path: /app/hr/api/v1/regions
```

## What Failed First

- **Symptom:** Generated `HrRegion` used `@Column(name = "region_id")` on the `@Id` field but the DB column is `REGION_ID` (uppercase style). Flyway migration failed.
- **Root cause:** Claude defaulted to lowercase column naming. Schema uses uppercase because it was designed for Vertex Tech conventions.
- **Fix:** Added `spring.jpa.hibernate.naming.physical-strategy=org.hibernate.boot.model.naming.PhysicalNamingStrategyStandardImpl` to application.yml and documented in CLAUDE.md.

- **Symptom:** `@SQLRestriction` import not found — Claude used the deprecated `@Where` annotation.
- **Root cause:** `@Where` was deprecated in Hibernate 6.2; `@SQLRestriction` is the replacement. Claude's training data included both.
- **Fix:** Added explicit import `import org.hibernate.annotations.SQLRestriction` and documented the correct annotation in CLAUDE.md.

## CLAUDE.md / Skill Update Made

```markdown
## Database Rules (additions)
- Use @SQLRestriction("deleted_at IS NULL") NOT @Where (deprecated in Hibernate 6.2)
- Naming strategy: PhysicalNamingStrategyStandardImpl — column names match schema exactly
```

Created skill `/scaffold-entity`:
```
Create a full entity scaffold following CLAUDE.md: Entity, Repository, DTO, Request,
Service (HrLogHelper), Controller (HrApiResponse). Entity name: $ENTITY_NAME,
table: $TABLE_NAME, fields: $FIELDS
```
**Why:** Eliminates re-explaining the scaffolding pattern for each of the 8 simple entities.

## Key Teaching Points

1. Skills compound engineering — define the pattern once, reuse it N times without context drift.
2. Deprecated annotations accumulate in training data — always specify the exact annotation + import in CLAUDE.md.
3. Physical naming strategy is non-obvious — document it before the first entity is built.
